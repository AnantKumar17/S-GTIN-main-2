# SGTIN Lifecycle System with Q&A Chatbot - Complete POC Roadmap

## Project Overview

**Title:** SGTIN Serialization with Intelligent Q&A Chatbot  
**Duration:** 48 hours  
**Team:** 3-5 developers  
**Approach:** Simplified chatbot (Q&A only), loosely coupled microservices, SAP-compatible stack

---

## Architecture Principles (CRITICAL FOR SUCCESS)

### 1. Simplified Chatbot Approach
- Chatbot ONLY answers questions - does NOT perform actions
- All CREATE/UPDATE/DELETE done via traditional UI
- Chatbot queries: "What's status of PO X?", "Show items without SGTIN", "List counterfeit scans"
- 6-8 hours to build vs 15+ for action-performing agent
- Still provides major differentiation

### 2. Loose Coupling for SAP Integration
- Each module = independent microservice
- Communication via REST APIs with OpenAPI specs
- No direct database coupling between services
- Clear boundaries: SGTIN Service, PO Service, Inventory Service, POS Service, Chatbot Service
- Easy to integrate into SAP S/4HANA as side-by-side extensions

### 3. SAP-Compatible Tech Stack
- **Backend:** Node.js with Express (SAP CAP compatible)
- **Database:** PostgreSQL (POC) to SAP HANA Cloud (Production)
- **API:** REST + OpenAPI 3.0 to OData V4 wrapper for production
- **Frontend:** React with SAP Fiori design tokens to UI5/Fiori Elements for production
- **Chatbot:** OpenAI GPT-4 / Claude with RAG (Retrieval Augmented Generation)
- **Auth:** JWT (POC) to SAP Cloud Identity (Production)

### 4. Multi-Tenancy from Day One
- All database tables include MANDT VARCHAR(3) field (SAP standard)
- Supports multiple clients/customers in single deployment
- Production-ready from POC stage

## Problem Statement (From Transcript)

### Core Problem
- Retail systems track products at SKU/GTIN level, not individual item level
- Know "we have 100 blue shirts" but NOT which exact physical units exist
- Blind spots: Which items sold, damaged, returned, stolen, or counterfeit?

### What SGTIN Solves
- Serialized GTIN assigns unique identity to every physical product unit
- Enables item-level tracking: Manufacturing to Warehouse to Store to POS to Customer
- Supports counterfeit detection, authenticity verification, precise recalls

### Business Impact
- Inventory shrinkage reduction
- Counterfeit detection at POS
- Unit-specific recalls (not batch-based)
- Warranty and authenticity automation

## Development Roadmap (48 Hours)

### Phase 0: Foundation Setup (1.5 hours)

**Task 0.1: Initialize Microservices Repository** (20 min)

Project structure:
- services/sgtin-service (SGTIN generation)
- services/po-service (Purchase Orders)
- services/inventory-service (Goods Receipt + Stock)
- services/pos-service (Sales Transactions)
- services/chatbot-service (Q&A Bot)
- frontend (React UI)
- database (Shared schema for POC only)
- docs (OpenAPI specs)

**Task 0.2: Setup SAP-Compatible Stack** (30 min)
- Node.js 18+ with Express
- PostgreSQL 14+ with HANA-compatible SQL
- React + Vite + ui5 webcomponents
- OpenAI/Claude API for chatbot
- Tools: bwip-js, qrcode, html5-qrcode

**Task 0.3: Design Service API Contracts** (40 min)

Create OpenAPI 3.0 specs for each service:
- SGTIN Service: POST /generate, GET /validate/{sgtin}
- PO Service: POST /purchase-orders, GET /purchase-orders/{id}
- Inventory Service: POST /goods-receipts, GET /inventory
- POS Service: POST /sales
- Chatbot Service: POST /chat/query

---

### Phase 1: Multi-Tenant Database (2 hours)

**Task 1.1: Design Database Schema** (50 min)

CRITICAL: All tables MUST include MANDT VARCHAR(3) for multi-tenancy

Key tables:
- products (mandt, gtin, name, category, brand)
- serialized_items (mandt, sgtin, gtin, status, location, batch, manufacture_date, passport)
- purchase_orders (mandt, po_id, gtin, quantity, status)
- po_sgtin_mapping (mandt, po_id, sgtin)
- goods_receipts (mandt, gr_id, po_id, warehouse, received_at)
- sales (mandt, sale_id, sgtin, store_id, sold_at)
- lifecycle_events (mandt, event_id, sgtin, event_type, metadata)

Indexes for chatbot queries:
- idx_sgtin_status ON serialized_items(mandt, status)
- idx_sgtin_batch ON serialized_items(mandt, batch)
- idx_po_status ON purchase_orders(mandt, status)

**Task 1.2: Create Node.js Data Models** (50 min)
- Base model class with MANDT handling
- Product, SerializedItem, PurchaseOrder, Sale models
- Use pg (PostgreSQL client)

**Task 1.3: Seed Sample Data** (20 min)
- 20 sample products (Adidas, Nike, H&M)
- Test batches: JAN26-ADIDAS-BLUE, etc.
- All data uses MANDT = '100'

---

### Phase 2: Core Microservices (5 hours)

**Task 2.1: SGTIN Generation Service** (70 min)
- Port: 3001
- Endpoints:
  - POST /api/sgtin/generate (Generate SGTINs in GS1 format)
  - GET /api/sgtin/validate/{sgtin} (Validate SGTIN)
- Logic: Generate 01{GTIN}21{Serial}, create lifecycle event
- Output: SGTINs + QR codes + barcodes

**Task 2.2: Purchase Order Service** (70 min)
- Port: 3002
- Endpoints:
  - POST /api/purchase-orders (Create PO + generate SGTINs)
  - GET /api/purchase-orders/{po_id} (Get PO details)
  - GET /api/purchase-orders?status={status} (List POs for chatbot)
- Integration: Calls SGTIN Service via HTTP

**Task 2.3: Inventory Service** (70 min)
- Port: 3003
- Endpoints:
  - POST /api/goods-receipts (Receive goods, update status to IN_STOCK)
  - GET /api/inventory (List all stock with filters)
  - GET /api/inventory/missing-sgtins (Products without serials for chatbot)
- Logic: Validate scanned SGTINs against PO

**Task 2.4: POS Service** (60 min)
- Port: 3004
- Endpoints:
  - POST /api/sales (Process sale, update status to SOLD)
  - GET /api/sales/{sale_id} (Get sale details)
- Counterfeit Detection: Reject unknown SGTINs, already-sold items

**Task 2.5: Traceability Engine** (50 min)
- Endpoint: GET /api/trace/{sgtin} (Full lifecycle timeline)
- Output: Created to Received to Sold with timestamps
